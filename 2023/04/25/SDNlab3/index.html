实验任务一：最小时延路径

该任务关键并不在于算法层面上如何求解最短路，基本框架本已实现好，需要理解并完成的是链路时延的计算。该计算主要依据 LLDP 以及 Echo 数据包。

当带有时间戳的 LLDP 经由交换机至 而后返回控制器时，控制器会根据接收时间以及发送时间计算出 控制器 控制器 的时延 lldp_delay_s12，同理可以计算 lldp_delay_s21。为获取 至 的时延，还需控制器至 以及 的时延，这步可经由 Echo 数据包实现。故可得链路平均时延为：

示例见的蓝色箭头。同理反向的时延由绿色的箭头组成。此外，控制器到交换机的往返时延由一个蓝色箭头和一个绿色箭头组成，此部分时延由echo报文测试

拓扑发现模块：

首先先判断src和dst的echo延迟是否被计算完成，再判当前链接的源端口号与 self.link_info 中存储的源端口号是否一致，如果一致，就计算delay，如果delay<0的话，意味着可能出现单向计算时延导致最后小于0，这是不允许的。则不进行更新，使用上一次原始值

首先发生echo报文，获取当前时间到datapath(不要同时发送echo请求，因为它几乎同时会生成大量echo回复 ，在echo_reply_处理程序中处理echo reply时，会产生大量队列等待延迟，我的程序因此出现了各种奇怪的 Bug ，包括但不限于：时而正常工作时而卡在发送 Echo Request 的过程中、根据 Echo Reply 计算的时延很大（很小）、始终不能向所有的交换机发出 Echo Request ，等等。不外乎是影响了消息处理和更新拓扑的协程正常工作。)

之后处理echo报文，得到echo_delay存入数据结构之中，就可以在更新拓扑时被使用来更新边权。计算最短路。

之后是一个_measure来产生线程，发送echo包。

  • MAIN_DISPATCHER: 触发器用于处理主要的 OpenFlow 消息，（Packet-In、Packet-Out、Flow-Mod 等）。

  • CONFIG_DISPATCHER: 触发器用于处理交换机的配置信息，（交换机的连接建立、端口状态变化、特性信息等）。

  • HANDSHAKE_DISPATCHER: 触发器用于处理与交换机之间的握手阶段相关的消息，（交换机的 Hello 消息、Features-Request 消息等）。

  • EventOFPEchoReply 在接收到交换机的 Echo Reply 消息时触发的事件类型，通过将其注册到 MAIN_DISPATCHER、CONFIG_DISPATCHER 和 HANDSHAKE_DISPATCHER 触发器中，可以在不同阶段处理不同类型的 Echo Reply 事件。

在计算完边权之后，就可以实现最短时延路径的计算，边权由delay给出，再由handle_ipv4函数调用shortest_path来获得最短路径，并下发流表给相应的交换机指导转发。

实验任务二：容忍链路故障

关键在于实现流表的删除。

对于每次link s1 s2 down 事件，都需要将相关路径上的交换机的流表删除。

在指导书上仅 SDC ping MIT 且只删除该路径上的某两个点的前提下，我一开始是如下的删除策略：

  • 查找路径上所有 datapath，按其中mac_to_port所有可达的 eth_dst进行 OFPMatch，进一步按此标准删除相关流表；

  • 查找路径上所有 datapath，按该路径起点 src_ip 以及终点 dst_ip进行 OFPMatch，进一步按此标准删除相关流表。

由于自始至终仅保留了一条 path，若 link down 的两个交换机都不在这条路上的话将删去零条流表项，以致 ping 不通的事件发生。实际上在实验中尝试了 pingall 后再 link s8 s9 down 就发生此类情况。

其实应该不只是src_ip和dst_ip那么简单，应该需要把所有涉及到这条路径的交换机路径的流表全部删除，而不单单只是这两个交换机，所以或许我们应该把所有的路径都保存起来，再进行检测（可模拟栈实现）

实际上，我的实现是删除所有交换机的ipv4和arp流表，流表项被删除后下一个数据包将导致 Table-miss ，进而导致 Packet-In，此时用最短路算法找跳数最少的路径并下发新的流表项。这似乎是一种最简单粗暴的方法。

首先定义delete_flow函数，删除流表项需要发送OFPFlowMod消息，指定command参数为ofproto.OFPFC_DELETE。需要注意的是，还必须指定out_port参数，否则消息可以被发送但删除并不会发生。这里直接令out_port=ofproto.OFPP_ANY表示删除任何满足条件的流表项。

在_port_status_handler函数中，匹配所有的ipv4和arp流表，并全部删除

(这里的优先级貌似没什么用，我一开始以为优先级比原来大才能删除，实验之后发现随便什么数字都可以，还有如果带上buffer_id，删除就会失败，总之这个OFPFlowMod规则较为严格)

由于链路状态改变，故 mac_to_port 以及 sw须更新，不然可能导致转发错误（直接将其清空）。

其他需要注意的点

  • network_awareness.py 中的_get_topology() 实现不太舒服，每次检测到拓扑结构改变我删去了旧图，否则将致误。可能是因为网络拓扑获得是经由 add_edge 实现的，所以不清空实际就会变成图中多了不少莫名其妙的边。
  • down掉之后重新ping，ping久了会发生广播风暴，这实际上是因为之前实现的arp_handler对这种情况不适用，需要修改

在之前的实验任务中，处理的是广播包，但在这个实验中，会有定时询问的单播包产生广播风暴，所以需要改变处理方式。
